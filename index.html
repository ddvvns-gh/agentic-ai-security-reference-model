What is depicted?

This model defines a target-state reference architecture for governing and securing enterprise AI agents as managed digital workers. It intentionally separates business governance, supply-chain provenance, machine-readable policy contracts, SDLC validation, runtime enforcement, human accountability, evidence, detection, resilience and assurance. The model is designed as a north-star architecture, not a claim that every control is mature or fully available today.

1. Govern before buildAgents require business purpose, owner, risk tier and approval before engineering begins.
2. Prodspec is the contractEvery runtime control consumes the same machine-readable policy contract.
3. Tool use is the action boundaryAgent risk materialises through tools, data access and external commitments.
4. Reduce blast radiusPrompt injection and semantic attacks are reduced through layered controls, not declared solved.
5. Evidence over assertionEvery capability must produce testable evidence and have a named owner.

How to read this architecture

This model is organised from top to bottom. It starts with business accountability and trust, then moves into engineering, runtime enforcement, decision governance, monitoring and resilience.

1 · GovernThe upper layers define ownership, approval, risk appetite, trusted components and the policy contract.
2 · EnforceThe middle layers translate policy into controls over identity, data, tools, SaaS, memory, cost and execution.
3 · AssureThe lower layers generate evidence, detect threats, support incident response and continuously test control effectiveness.

Policies flow downward. Telemetry, evidence and assurance flow upward. Together these layers create a secure, governed, observable and accountable operating model for enterprise-scale AI agents.

Layer -2: Prodspec Control Contract (Machine-Readable Policy Anchor)
L-2

Prodspec Control Contract

Creates a machine-readable contract defining how an AI agent may behave, what it can access and what it can do.

OutcomeTurns governance intent into enforceable policy that runtime controls can consume automatically.
AI-SPM / Policy
Gap
08
Agentic Prodspec Control Contract
Function: Machine-readable contract binding agent identity, owner, purpose, risk tier, models, prompts, data, tools, memory, approvals, costs and fail-safe rules.
Problem: A diagram or inventory does not enforce behaviour unless translated into machine-readable policy.
Control Pattern: Make the Prodspec the single control contract consumed by CI/CD, gateways, identity, policy engines, observability and evidence stores.
Evidence Hook: Prodspec schema; policy mappings; approved versions; enforcement logs.
Tool Registry
Gap
09
Tool Invocation Policy Model
Function: Defines allowed tools, actions, parameters, destinations, transaction classes, output commitment levels and approval triggers.
Problem: Tool use is the true action boundary for agents, but it is often governed too coarsely.
Control Pattern: Apply action-level authorization, schema validation, parameter constraints, egress checks and dual-control for high-impact tools.
Evidence Hook: Tool registry; policy decisions; blocked calls; approval records.
A2A Trust
Gap
10
Agent-to-Agent Communication Governance
Function: Controls peer-agent and parent-child communication, message provenance, delegation, trust boundaries and protocol security.
Problem: Multi-agent systems can transfer poisoned context, spoof authority or cascade failures.
Control Pattern: Use authenticated messages, signed context, scoped delegation, message provenance and isolation between trust domains.
Evidence Hook: Message provenance; mTLS/token records; delegation chain; blocked peer calls.
Privacy / Records
Maturing
11
Data Residency, Privacy & Records Policy
Function: Defines privacy impact, retention, jurisdiction, customer-data handling, telemetry capture and records obligations.
Problem: Memory, prompts, logs and evidence records can create privacy and records-retention exposure.
Control Pattern: Bind data residency, retention, minimisation, deletion and audit retention requirements to the Prodspec.
Evidence Hook: PIA/DPIA; records schedule; retention logs; deletion evidence.
↓ Policy, evidence and control context flow downward; telemetry and assurance flow upward ↓
Layer -1: SDLC & Pre-Deployment Validation
L-1

SDLC & Pre-Deployment Validation

Validates AI systems before production through testing, assurance and approval controls.

OutcomeEnsures only tested, approved and evidenced AI systems reach production.
CI/CD / GitOps
Existing
12
Prompt & Use-Case Lifecycle Management
Function: Treats prompts, model configurations and agent logic as code with risk-tiered approval workflows.
Problem: Untracked prompt changes and uncontrolled use cases can change production behaviour without governance.
Control Pattern: Use version control, peer review, risk-tiered approvals and controlled promotion across environments.
Evidence Hook: PRs; approvals; change history; release evidence.
AI SecOps
Maturing
13
Comprehensive Testing Regime
Function: Runs red teaming, regression, jailbreak, data leakage, bias, tool misuse and resilience tests.
Problem: Latent model, prompt and tool flaws may not appear until production usage.
Control Pattern: Require signed evidence of test execution, results, exceptions and risk acceptance before promotion.
Evidence Hook: Test reports; exception records; signed evidence; regression baseline.
Shadow / AIOps
Emerging
14
Shadow Validation Environment
Function: Validates agent behaviour using synthetic, masked, replayed or explicitly approved read-only production data with no state-changing execution.
Problem: Agents may behave differently with real enterprise messiness, but live-data shadowing can create privacy risk.
Control Pattern: Use controlled shadow evaluation, simulated tools and monitored read-only access for approved scenarios.
Evidence Hook: Shadow run reports; dataset approval; no-write evidence; behaviour deltas.
Release Control
Maturing

Production Readiness & Release Gate
Function: Confirms all required supply-chain, testing, Prodspec, ownership, risk and evidence gates are satisfied before deployment.
Problem: A model can pass tests while still missing ownership, risk acceptance or control integration.
Control Pattern: Use a release gate that blocks deployment unless all mandatory controls and approvals are present.
Evidence Hook: Release checklist; automated gate logs; risk acceptance; deployment manifest.

Existing

Managed Endpoint Enforcement
Function: Controls local AI interactions, browser sessions, clipboard/download behaviour and unauthorized process execution.

Identity Auth

Function: Replaces static credentials with short-lived, policy-derived, least-privilege credentials that cannot exceed approved scope.

Layer 2: Runtime, Data, Tool, SaaS, Memory & Cost Enforcement
L2

Runtime, Data, Tool, SaaS, Memory & Cost Enforcement

Controls what AI agents can do while operating, including data access, tools, SaaS features, memory and cost.

OutcomeKeeps agents within defined security, privacy, operational and economic boundaries.
AI Firewall
Existing / Limited
25
Prompt & Output Inspection
Function: Inspects prompts and outputs to reduce unsafe content, injection exposure, malicious code, policy violations and data leakage.
Problem: Prompt injection and unsafe output cannot be fully prevented by a single semantic firewall.
Control Pattern: Use inspection as one layer alongside least privilege, tool controls, provenance, DLP, HITL and blast-radius limits.
Evidence Hook: Inspection logs; false-positive metrics; blocked outputs; bypass analysis.
DSPM / DLP / RAG
Maturing
26
Data Boundary & RAG Governance
Function: Enforces data classification, ingestion provenance, retrieval authorization, vector-store controls, egress checks and deletion requirements.
Problem: Agents can retrieve, infer, persist or export sensitive data beyond intended boundaries.
Control Pattern: Apply data policy at ingestion, retrieval, memory write, tool use and egress points.
Evidence Hook: Classification tags; retrieval decisions; DLP logs; vector-store lineage.
SSPM / SaaS AI
Maturing
27
Third-Party SaaS AI Governance
Function: Manages native SaaS copilots, tenant settings, opt-out controls, data boundary settings, logging and conditional access.
Problem: SaaS AI features can expose data through opaque platform behaviour or weak tenant controls.
Control Pattern: Continuously assess and enforce SaaS AI configurations, logging, data-use restrictions and access policy.
Evidence Hook: SSPM findings; SaaS audit logs; settings evidence; exception records.
Gateway FinOps
Maturing
28
FinOps & Runtime Cost Controls
Function: Applies token, API, model, workflow and lineage-level budgets, quotas and rate limits.
Problem: Runaway agents can create economic denial of service and uncontrolled cloud consumption.
Control Pattern: Enforce budgets, loop limits, retry caps and anomaly alerts; terminate or throttle where thresholds are exceeded.
Evidence Hook: Cost dashboards; quota events; throttling logs; anomaly alerts.
Memory Gate
Gap
29
Agent Memory Governance
Function: Controls persistent and session memory reads/writes, retention, provenance, poisoning risk, deletion and re-scoring.
Problem: Long-lived memory can retain sensitive data or accumulate poisoned instructions.
Control Pattern: Classify memory entries, restrict writes, expire stale data, quarantine risky context and support deletion obligations.
Evidence Hook: Memory ledger; write decisions; expiry logs; quarantine events.
Commitment Gate
Gap
30
Output Commitment Control
Function: Distinguishes draft, recommendation, notification, API call, transaction, customer decision and irreversible external commitment.
Problem: The same model output can be harmless as a draft but high-risk if automatically executed.
Control Pattern: Apply commitment-level policy, HITL triggers, dual control and transaction signing before external or irreversible action.
Evidence Hook: Commitment records; approval logs; signed transactions; blocked actions.

Threshold Routing Matrix: Deterministic, Probabilistic & High-Impact Response
TRM

Threshold Routing Matrix

Determines how the organisation responds when AI activity exceeds approved thresholds or enters uncertain territory.

OutcomeProvides consistent risk-based responses: contain hard breaches, evaluate ambiguity, and escalate high-impact actions.
Infrastructure Enforcer
Maturing
31
Terminate, Contain & Compensate - Hard
Function: Revokes tokens, terminates processes, quarantines state and rolls back only where technically reversible.
Problem: Some violations need immediate containment, but not all actions can be rolled back.
Control Pattern: Use terminate, revoke, isolate, compensate or rollback based on action reversibility and dependency map.
Evidence Hook: Containment logs; revocation records; rollback/compensation evidence.
Logic Governor
Gap
32
Yield & Evaluate - Soft
Function: Routes ambiguous semantic deviations to deterministic policy checks and evaluator review without automatically changing policy boundaries.
Problem: Soft drift creates false positives if hard-blocked but risk if blindly allowed.
Control Pattern: Allow or deny the specific action; policy boundary changes require approved policy update and redeployment.
Evidence Hook: Evaluator record; policy decision; exception workflow; drift trend.
Intent Validator
Maturing
33
Pause & Escalate - High Impact
Function: Freezes high-impact or irreversible actions until authorized approvers complete required review.
Problem: Irreversible or regulated actions require accountability before execution.
Control Pattern: Trigger HITL, dual control, data owner approval, change approval or risk acceptance based on action class.
Evidence Hook: Approval evidence; paused action record; escalation trail.

Layer 3: Cognitive Governance, Human Accountability & Evidence

Maturing

Human Accountability & Dual Control
Function: Verifies authorised accountable humans or dual-control approvers for high-risk action classes.
Problem: Too many approvals cause rubber-stamping; too few leave accountability gaps.

Function: Runs deterministic policy-as-code first, then evaluator models for contextual assessment where appropriate.
Problem: LLM-only governance is non-deterministic; rule-only governance misses nuance.
Control Pattern: Use deterministic enforcement for hard rules and evaluator assistance for recommendations or ambiguity.
Evidence Hook: OPA/Rego decisions; evaluator results; calibration data; overrides.
Audit Ledger
Gap

Function: Creates signed evidence linking intent, Prodspec, prompt/model versions, retrieved context, policy decisions, tools, approvals, output and execution payload.
Problem: Auditability is not the same as proving internal model reasoning.
Control Pattern: Record evidence chain and explainability artefacts without claiming full proof of causal reasoning.
Evidence Hook: Evidence record; signatures; model/prompt versions; retrieval references.

Function: Supports customer-impacting decision appeal, human override, temporary exceptions, compensating controls and expiry.
Problem: Regulated decisions need recourse, and exceptions can become permanent if unmanaged.
Control Pattern: Require override authority, exception expiry, compensating controls, impact review and audit evidence.
Evidence Hook: Appeal records; exception register; expiry logs; compensating controls.

Layer 4: Observability, Incident Response, Assurance & Resilience
L4

Observability, Incident Response, Assurance & Resilience

Monitors AI behaviour, detects threats, manages incidents and validates control effectiveness over time.

OutcomeCreates continuous visibility, rapid response and long-term operational resilience.
LLM Observability
Maturing
38
Structured Agentic Telemetry Collection
Function: Captures prompts, context references, tool calls, memory writes, policy decisions, token usage, approvals, errors and security summaries.
Problem: Traditional logs do not capture agent behaviour, but storing chain-of-thought can be unsafe and unreliable.
Control Pattern: Capture structured telemetry and summaries rather than relying on true internal reasoning traces.
Evidence Hook: Telemetry schema; trace IDs; log retention; privacy controls.
XDR / SIEM / SOAR
Maturing
39
Correlated AI Threat Detection & IR
Function: Correlates agent telemetry with endpoint, identity, network, data and application logs, then triggers AI-specific incident playbooks.
Problem: Prompt injection, rogue agents and data leakage require coordinated containment and forensics.
Control Pattern: Build playbooks for rogue agent, prompt injection, memory poisoning, data leakage, cost runaway and supply-chain compromise.
Evidence Hook: SIEM alerts; SOAR playbooks; incident records; forensic snapshots.
AI GRC Monitor
Gap
40
Bias, Drift & Control Effectiveness Monitoring
Function: Monitors systemic bias, model drift, performance degradation and control effectiveness against defined thresholds.
Problem: Agents can degrade or create unfair outcomes over time even if individual actions seem acceptable.
Control Pattern: Use continuous evaluation, sample review, fairness metrics, drift tests and control test evidence.
Evidence Hook: Drift reports; fairness metrics; sample reviews; control test results.
Resilience Map
Gap
41
Resilience Dependency & Fallback Management
Function: Defines fail-open, fail-closed, degrade, queue, manual takeover and approved model/provider fallback behaviour.
Problem: Failure of gateway, policy engine, model provider, DLP or observability can create outages or unsafe bypasses.
Control Pattern: Bind fallback posture to risk tier and action class, and test resilience scenarios regularly.
Evidence Hook: Dependency map; fail-state tests; fallback logs; recovery evidence.

Gap
