Inspect prompts and outputs as advisory detection, not sole enforcement.

This capability inspects prompts, retrieved context and outputs for injection, leakage and unsafe content, but explicitly recognises that semantic inspection is not a deterministic security boundary. It feeds external enforcement, routing and evidence.

Purpose and strategic objective

Purpose

To reduce runtime risk from malicious prompts, poisoned retrievals and unsafe outputs while avoiding false confidence in model-level self-policing.

Strategic objective

Use inspection as a detection and routing signal, combined with deterministic identity, tool, data and output-commitment controls.

Architectural placement

Why this sits here

This sits in Layer 2 because inspection occurs during live interactions and influences enforcement pathways.

Prompt inspection helps, but it cannot fully solve instruction/data separation inside the LLM context window.
Problem statement

LLMs process instructions and data in the same context. There is no complete mathematical separation that makes prompt injection impossible. Therefore inspection must be layered with external tool, data and identity controls.

Capability description

What the capability does

  • Inspects direct prompts, retrieved context and outputs.
  • Classifies suspected injection, leakage and unsafe content.
  • Feeds tool gateway, output commitment and incident-response decisions.
  • Captures inspection evidence for tuning and assurance.
  • Escalates uncertain cases to dual-engine or human review.

What the capability does not do

  • It does not make the LLM a security boundary.
  • It does not fully prevent prompt injection.
  • It does not replace deterministic tool, identity, sandbox and data controls.

v18 hardening — Semantic Payload Inspection

Runtime inspection now differentiates between user prompts, retrieved context, agent-to-agent payloads, memory recalls and final outputs. Each class requires different inspection, provenance and routing logic. Inspection remains probabilistic and must feed external deterministic controls rather than becoming the only enforcement mechanism.

Enterprise examples

RAG document attack

A document contains hidden instructions.

Normal / desired state:
Inspection flags risk and tool gateway restricts action.
Risk / control concern:
Agent is merely reminded to ignore malicious text.

Customer email

Agent drafts external response.

Normal / desired state:
Output is inspected and routed to commitment control.
Risk / control concern:
Model self-certifies that output is safe.
Failure scenarios

Self-policing bypass

Agent says content is safe and proceeds.

Indirect injection bypass

Poisoned context manipulates tool parameters.

Sensitive output

Agent includes confidential content in public response.
