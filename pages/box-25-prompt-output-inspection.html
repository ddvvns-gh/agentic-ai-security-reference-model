Capability definition

Prompt & Output Inspection is a reference-model capability within Agent Runtime Security. Inspects prompts and outputs to reduce unsafe content, injection exposure, malicious code, policy violations and data leakage.

This page expands the box into a practical reference model view: placement, purpose, risks, control objectives, examples, ownership, metrics, technology patterns and maturity guidance.

Placement and intent

Where this sits in the model

Layer 2: Runtime, Data, Tool, SaaS, Memory & Cost Enforcement

This placement is intentional because the capability governs the point where business intent, technical execution, policy, evidence and risk intersect for this part of the agentic AI operating model.

Enterprise governance / policy ↓ Layer 2: Runtime, Data, Tool, SaaS, Memory & Cost Enforcement ↓ Prompt & Output Inspection ↓ Control evidence, telemetry and improvement signals

Why this box exists

Prompt injection and unsafe output cannot be fully prevented by a single semantic firewall.

Architectural intent: Use inspection as one layer alongside least privilege, tool controls, provenance, DLP, HITL and blast-radius limits.
Purpose and strategic objectives

Purpose

Provide a named, governable and testable capability that reduces the relevant enterprise risk while enabling safe adoption of agentic AI.

The purpose is to make this control area visible, owned, measurable and enforceable, rather than leaving it as an implied responsibility spread across teams.

Strategic objectives

  1. Establish clear ownership and accountability.
  2. Define control objectives and measurable evidence.
  3. Reduce inherent risk through enforceable patterns.
  4. Support executive, risk and audit reporting.
  5. Enable safe, scalable enterprise adoption of AI agents.
  6. Provide a foundation for detailed implementation standards.

Control objectives

Minimum control objectives

  • CO-01: Scope, ownership and accountability must be explicitly defined.
  • CO-02: The capability must be represented in the relevant inventory or Prodspec where applicable.
  • CO-03: Usage must be risk-assessed before production or enterprise enablement.
  • CO-04: Controls must be testable and generate evidence.
  • CO-05: Exceptions must have owner, expiry, risk acceptance and compensating controls.
  • CO-06: Changes must trigger revalidation where risk materially changes.

What this capability does not do

  • It does not replace enterprise risk management.
  • It does not guarantee zero risk or perfect prevention.
  • It does not remove the need for accountable owners.
  • It does not define a single mandatory vendor product.
  • It does not remove the need for control testing and assurance.

Real enterprise examples

Generated code execution

An agent produces a script. The capability routes execution to approved sandbox or blocks local execution.

Tool call

An agent attempts a tool call outside its Prodspec. The capability rejects or escalates the action.

Cost loop

An agent enters a recursive loop. Runtime quotas and lineage budgets stop uncontrolled consumption.

Failure scenario

Example failure mode

Prompt injection and unsafe output cannot be fully prevented by a single semantic firewall.

Trigger / change detected ↓ Risk and policy evaluation ↓ Contain, restrict, approve or escalate ↓ Evidence captured ↓ Review and improve control

Required response

  • Identify affected agents, users, tools, data, systems or decisions.
  • Determine whether containment, restriction, escalation or risk acceptance is required.
  • Capture evidence for audit and post-incident review.
  • Update Prodspec, policy, test suite or operating procedure where necessary.

Implementation notes

  • Start with inventory, ownership and policy before automation.
  • Prioritise high-risk, customer-facing, privileged and data-intensive agents.
  • Integrate with CI/CD, IAM, data governance, SIEM and GRC where relevant.
  • Define evidence requirements before declaring the control operational.
Regulatory alignment

This capability can support evidence and control narratives for APRA CPS 230, APRA CPS 234, Privacy Act 1988 / OAIC APPs, NIST AI RMF, ISO/IEC 42001, ISO/IEC 27001 and the EU AI Act where applicable. Exact mapping should be performed in the control catalogue.
