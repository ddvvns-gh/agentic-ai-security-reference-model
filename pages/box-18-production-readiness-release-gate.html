This capability provides the final release decision point. It confirms the agent has an approved Prodspec, tested artefacts, ownership, risk acceptance, supply-chain evidence, runtime integration and operational readiness before deployment.

Purpose and strategic objective

Purpose

To ensure AI agents cannot enter production unless required governance, security, risk and operational criteria are met.

Strategic objective

Create a defensible release process that prevents unapproved, untested or unevidenced AI systems from becoming production services.

Architectural placement

Why this sits here

It sits in Layer -1 because it is the final pre-production gate before runtime operation.

The release gate converts many separate controls into a single go/no-go deployment decision.
Problem statement

AI systems can appear ready while still missing ownership, test evidence, data approval, identity integration, monitoring, cost limits or incident response procedures.

Capability description

What the capability does

  • Checks Prodspec approval and version.
  • Verifies required tests and exceptions.
  • Confirms artefact, registry and BOM evidence.
  • Confirms identity, runtime and telemetry integration.
  • Blocks deployment when mandatory evidence is missing.

What the capability does not do

  • It does not replace individual controls.
  • It does not guarantee future runtime behaviour.
  • It does not remove the need for post-deployment monitoring.

Enterprise examples

High-risk agent launch

A customer-facing agent is ready for release.

Normal / desired state:
Release gate confirms test, risk, data, identity and telemetry evidence.
Risk / control concern:
The agent launches with missing incident response integration.

Emergency AI change

A critical prompt fix is required.

Normal / desired state:
Emergency change path captures approval and post-release review.
Risk / control concern:
A direct production change bypasses testing and evidence.
Failure scenarios

Missing Prodspec

Deployment proceeds without a valid policy contract.

Skipped testing evidence

Release is approved based on demonstration rather than test report.

No monitoring integration

SOC cannot see the agent after go-live.

North-star vision

Every production AI agent is released through a risk-tiered gate that confirms ownership, policy, testing, evidence and operational readiness.

Box 33 full-depth alignment

Full-depth update: This Prodspec Control Contract capability is upgraded to define enforceable control obligations, measurable evidence and runtime traceability before agentic execution.

Control objective

Ensure the capability defines clear policy intent, technical control requirements, evidence obligations and threshold behaviour for all material AI systems that depend on this contract.

Control statement

Agentic AI systems must not rely on this capability unless the prodspec contract is versioned, approved, testable, monitored and linked to runtime evidence.

Risk catalogue

  • R-PSC-001: Prodspec contract is ambiguous or unenforceable.
  • R-PSC-002: Runtime behaviour deviates from declared contract.
  • R-PSC-003: Contract changes are not versioned, approved or tested.
  • R-PSC-004: Evidence required by contract is not produced or retained.

Threat events

  • Agent executes an action not covered by approved control contract.
  • Contract update changes permissible behaviour without revalidation.
  • Evidence collection fails during material decision or tool invocation.
  • Threshold escalation is bypassed because contract logic is incomplete.

Preventive controls

  • Contract versioning and approval.
  • Policy-as-code validation.
  • Threshold and evidence requirement declaration.
  • Runtime gating on contract validity.

Detective / corrective controls

  • Contract conformance monitoring.
  • Runtime drift detection.
  • Exception and waiver workflow.
  • Rollback to last approved contract version.

Metrics

  • Approved contract coverage = 100% for material systems.
  • Contract conformance rate ≥ 99%.
  • Unapproved contract changes = 0.
  • Evidence completeness = 100% for T3–T5 workflows.

Evidence

  • Approved prodspec contract.
  • Version history and approval record.
  • Policy-as-code test results.
  • Runtime conformance and evidence logs.

Traceability

Risk: runtime behaviour deviates from contract → Threshold: material AI workflow → Control: contract validity and conformance gate → Metric: conformance rate → Evidence: approved contract and runtime conformance logs.
