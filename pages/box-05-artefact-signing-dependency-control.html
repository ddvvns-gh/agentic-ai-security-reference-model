Verify the integrity of AI artefacts and dependencies before they are used.

This capability applies software supply-chain discipline to AI artefacts: model files, containers, prompts, embeddings, deployment manifests, packages and agent frameworks. It helps ensure what is deployed is what was approved.

Purpose and strategic objective

Purpose

To protect AI systems from tampered artefacts, dependency compromise and unverified build outputs.

Strategic objective

Ensure deployed AI artefacts are authentic, approved, versioned and traceable to a trusted build and registry.

Architectural placement

Why this sits here

It sits in Layer -3 because artefact integrity must be established before engineering and runtime enforcement can rely on anything.

Signed and verified artefacts create the trust foundation that deployment gates and runtime controls depend on.
Problem statement

AI artefacts are increasingly assembled from many components. A single compromised dependency, model file, prompt package or container image can undermine the entire control model.

Capability description

What the capability does

  • Signs model artefacts, containers, prompts and manifests.
  • Verifies signatures during build and deployment.
  • Tracks dependency versions and known vulnerabilities.
  • Blocks unsigned or unapproved artefacts.
  • Supports SBOM, AI-BOM and ML-BOM evidence.

What the capability does not do

  • It does not guarantee model behaviour is safe.
  • It does not replace testing or runtime monitoring.
  • It does not remove the need for third-party risk review.

Enterprise examples

Model artefact release

A model is promoted to production.

Normal / desired state:
The model hash and signature match the approved registry record.
Risk / control concern:
A modified model file is deployed outside the approved release path.

Package compromise

A commonly used dependency is reported compromised.

Normal / desired state:
The BOM identifies affected agents and triggers revalidation.
Risk / control concern:
The enterprise cannot identify which agents use the dependency.
Failure scenarios

Unsigned prompt package

A prompt bundle is changed without review.

Compromised package update

A dependency update introduces malicious behaviour.

Deployment manifest mismatch

The deployed configuration differs from the approved Prodspec.

North-star vision

Every deployed AI artefact is signed, verified and traceable to an approved build and dependency baseline.

Box 30 full-depth alignment

Full-depth update: Artifact signing and dependency control now extends to model weights, prompts, tool manifests, policy files, container images, orchestration code and generated agent packages.

Control objective

Ensure AI artifacts and dependencies are tamper-evident, approved and traceable from source through deployment and runtime execution.

Control statement

Material AI artifacts must be signed, verified and bound to a dependency manifest before execution. Unsigned, tampered or unapproved artifacts must be blocked from production use.

Risk catalogue

  • R-SCP-005: Tampered artifact enters production.
  • R-SCP-006: Dependency substitution or version confusion.
  • R-SCP-007: Prompt or policy asset altered outside approval path.
  • R-SCP-008: Runtime executes artifact that does not match approved manifest.

Threat events

  • Prompt template modified in repository without approval.
  • Model artifact hash differs from registered version.
  • Tool manifest changed after certification.
  • Dependency package replaced by malicious or vulnerable version.

Preventive controls

  • Artifact signing and hash verification.
  • Dependency lock files and signed manifests.
  • Policy-as-code approval gates.
  • Deployment block on signature mismatch.

Detective / corrective controls

  • Signature verification monitoring.
  • Manifest drift detection.
  • Artifact quarantine.
  • Rollback to last approved signed version.

Metrics

  • Signed artifact coverage = 100% for production AI artifacts.
  • Manifest verification success ≥ 99%.
  • Unauthorized artifact execution = 0.
  • Mean time to quarantine tampered artifact within SLA.

Evidence

  • Signed artifact manifest.
  • Hash verification logs.
  • Deployment approval record.
  • Quarantine and rollback records.

Traceability

Risk: Tampered artifact execution → Threshold: production artifact → Control: signature and manifest verification → Metric: verification success rate → Evidence: signed manifest and verification log.
