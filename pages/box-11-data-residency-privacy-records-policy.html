Bind AI usage to jurisdiction, privacy, retention and records obligations.

This capability defines how prompts, outputs, telemetry, memories, retrieved context and evidence records must be handled according to privacy, retention, data residency and records-management requirements.

Purpose and strategic objective

Purpose

To ensure AI agents do not create unmanaged privacy, jurisdiction or records-retention exposure.

Strategic objective

Make data handling obligations explicit and enforceable before AI agents process, store, remember, log or transmit sensitive information.

Architectural placement

Why this sits here

It sits in Layer -2 because data and records obligations must be encoded into the Prodspec before engineering and runtime decisions occur.

AI creates new records: prompts, outputs, memories, traces and evidence. These must be governed from the start.
Problem statement

AI systems may process sensitive information in prompts, retrieved context, memory stores, telemetry logs and decision evidence. If retention, minimisation and jurisdiction are not designed upfront, the enterprise may create privacy and records risk unintentionally.

Capability description

What the capability does

  • Defines residency and jurisdiction constraints.
  • Specifies prompt, output, memory and telemetry retention rules.
  • Sets minimisation, deletion and masking requirements.
  • Links data classification to AI processing rules.
  • Supports privacy impact and records evidence.

What the capability does not do

  • It does not replace enterprise data governance.
  • It does not guarantee provider compliance without contractual controls.
  • It does not remove the need for runtime DLP and monitoring.

Enterprise examples

Customer-support agent

An agent processes customer case notes.

Normal / desired state:
Prompts and outputs follow approved retention and masking rules.
Risk / control concern:
Sensitive prompts are retained indefinitely by a provider.

Audit evidence capture

Decision records are stored for regulatory evidence.

Normal / desired state:
Evidence is retained under defined records schedule.
Risk / control concern:
Evidence contains excessive personal data and no deletion path.
Failure scenarios

Unapproved region processing

A provider routes inference through an unapproved geography.

Persistent sensitive memory

An agent stores personal information in long-term memory without consent.

Telemetry overcollection

Logs capture full prompts unnecessarily.

Control objective

Ensure the capability defines clear policy intent, technical control requirements, evidence obligations and threshold behaviour for all material AI systems that depend on this contract.

Control statement

Agentic AI systems must not rely on this capability unless the prodspec contract is versioned, approved, testable, monitored and linked to runtime evidence.

Risk catalogue

  • R-PSC-001: Prodspec contract is ambiguous or unenforceable.
  • R-PSC-002: Runtime behaviour deviates from declared contract.
  • R-PSC-003: Contract changes are not versioned, approved or tested.
  • R-PSC-004: Evidence required by contract is not produced or retained.

Threat events

  • Agent executes an action not covered by approved control contract.
  • Contract update changes permissible behaviour without revalidation.
  • Evidence collection fails during material decision or tool invocation.
  • Threshold escalation is bypassed because contract logic is incomplete.

Preventive controls

  • Contract versioning and approval.
  • Policy-as-code validation.
  • Threshold and evidence requirement declaration.
  • Runtime gating on contract validity.

Detective / corrective controls

  • Contract conformance monitoring.
  • Runtime drift detection.
  • Exception and waiver workflow.
  • Rollback to last approved contract version.

Metrics

  • Approved contract coverage = 100% for material systems.
  • Contract conformance rate ≥ 99%.
  • Unapproved contract changes = 0.
  • Evidence completeness = 100% for T3–T5 workflows.

Evidence

  • Approved prodspec contract.
  • Version history and approval record.
  • Policy-as-code test results.
  • Runtime conformance and evidence logs.

Traceability

Risk: runtime behaviour deviates from contract → Threshold: material AI workflow → Control: contract validity and conformance gate → Metric: conformance rate → Evidence: approved contract and runtime conformance logs.
