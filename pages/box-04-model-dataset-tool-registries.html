Establish trusted provenance, approval chaining and supply-chain assurance for every AI artefact.

This capability establishes the enterprise trust chain for models, datasets, prompts, agents, tools, MCP servers, evaluators, memory stores, SaaS AI capabilities and AI-generated artefacts. It chains provenance, ownership, approval, dependency mapping and cryptographic attestation before artefacts are used by agents.

Purpose and strategic objective

Purpose

To maintain provenance, approval, dependency and attestation records for every AI artefact so agents are built and operated only from trusted, governed and continuously revalidated components.

Strategic objective

Create a verifiable AI trust chain by linking artefact origin, ownership, approval, dependencies, cryptographic integrity and ongoing assurance status.

Architectural placement

Why this sits here

It sits in Layer -3 because provenance and supply-chain assurance establish foundational trust before any agent is designed, built, deployed or delegated authority.

If artefact trust is weak, identity, execution and decision controls inherit a compromised foundation.
Problem statement

AI systems depend on models, datasets, prompts, tools and frameworks that may come from internal teams, open source, SaaS providers or third parties. Without controlled registries, the enterprise cannot reliably determine what is approved, trusted, licensed or safe to use.

Capability description

What the capability does

  • Maintains approved model, dataset, prompt and tool registries.
  • Records provenance, licence, classification, owner and approval status.
  • Blocks or flags unapproved components in engineering pipelines.
  • Supports revalidation when components change.
  • Feeds the Prodspec and AI-BOM.

What the capability does not do

  • It does not prove a model is safe in all conditions.
  • It does not replace legal, privacy or security review.
  • It does not perform runtime monitoring by itself.

v18 capability expansion — AI Provenance & Supply Chain Assurance

This capability now governs the full AI supply chain, not only registries. It chains provenance and approval so the enterprise can answer: what is this artefact, where did it come from, who or what created it, who approved it, what depends on it, has it changed, and can it still be trusted?

ModelsDatasetsPromptsAgentsTools / MCPEvaluatorsMemory StoresSaaS AIHuman-created ArtefactsAI-generated Artefacts

  • Provenance: origin, creator, generator, source, licence, data lineage and version lineage.
  • Approval chain: owner, reviewer, risk status, policy status, approval status and revalidation status.
  • Dependency chain: which agents, prompts, tools, datasets, policies, memory stores and SaaS AI features consume the artefact.
  • Cryptographic assurance: signatures, hashes, attestations, tamper-evident records and verification before use.
  • Continuous assurance: revalidation when artefacts change, dependencies become vulnerable, source trust changes or AI-generated artefacts are introduced.

Outcomes delivered

Trusted component baseline

Engineering teams start from approved, known sources.

Supply-chain visibility

The enterprise knows which artefacts are used by which agents.

Faster approvals

Pre-approved registries reduce repeated review effort.

Enterprise examples

Open-source model adoption

A team wants to use a new open-source model.

Normal / desired state:
The model is registered, scanned, licensed and approved before use.
Risk / control concern:
The model is downloaded directly and embedded into a production workflow.

New agent plugin

A vendor releases a plugin for workflow automation.

Normal / desired state:
The plugin is reviewed and added to the approved tool registry.
Risk / control concern:
The plugin is enabled without knowing its permissions or data handling.
Failure scenarios

Unknown model lineage

An agent relies on a model with unclear origin or licence.

Poisoned dataset reuse

A dataset from an unverified source is reused across multiple agents.

Unapproved plugin activation

A tool is enabled before security and data owners understand its access scope.
