Create approved registries for the components that AI agents are allowed to use.

This capability establishes controlled entry points for models, datasets, prompts, APIs, plugins, tools and agent frameworks. It prevents untrusted AI components from entering the enterprise environment without provenance, approval, classification and ownership.

Purpose and strategic objective

Purpose

To maintain approved registries for AI components so engineering teams can build only from known, trusted and governed sources.

Strategic objective

Reduce supply-chain risk by ensuring all AI artefacts have provenance, trust metadata, ownership and approval status before use.

Architectural placement

Why this sits here

It sits in Layer -3 because registries establish foundational trust before any agent is designed, built or deployed.

If the ingredients are untrusted, every downstream control starts from a compromised position.
Problem statement

AI systems depend on models, datasets, prompts, tools and frameworks that may come from internal teams, open source, SaaS providers or third parties. Without controlled registries, the enterprise cannot reliably determine what is approved, trusted, licensed or safe to use.

Capability description

What the capability does

  • Maintains approved model, dataset, prompt and tool registries.
  • Records provenance, licence, classification, owner and approval status.
  • Blocks or flags unapproved components in engineering pipelines.
  • Supports revalidation when components change.
  • Feeds the Prodspec and AI-BOM.

What the capability does not do

  • It does not prove a model is safe in all conditions.
  • It does not replace legal, privacy or security review.
  • It does not perform runtime monitoring by itself.

Outcomes delivered

Trusted component baseline

Engineering teams start from approved, known sources.

Supply-chain visibility

The enterprise knows which artefacts are used by which agents.

Faster approvals

Pre-approved registries reduce repeated review effort.

Enterprise examples

Open-source model adoption

A team wants to use a new open-source model.

Normal / desired state:
The model is registered, scanned, licensed and approved before use.
Risk / control concern:
The model is downloaded directly and embedded into a production workflow.

New agent plugin

A vendor releases a plugin for workflow automation.

Normal / desired state:
The plugin is reviewed and added to the approved tool registry.
Risk / control concern:
The plugin is enabled without knowing its permissions or data handling.
Failure scenarios

Unknown model lineage

An agent relies on a model with unclear origin or licence.

Poisoned dataset reuse

A dataset from an unverified source is reused across multiple agents.

Unapproved plugin activation

A tool is enabled before security and data owners understand its access scope.
