Control what agents can remember, retrieve, expire and forget.

This capability governs session and persistent memory, including write permissions, provenance, classification, retention, poisoning detection, deletion and re-scoring of memory entries.

Purpose and strategic objective

Purpose

To prevent agent memory from becoming an unmanaged store of sensitive data, poisoned instructions or stale context.

Strategic objective

Enable useful personalisation and continuity while preserving privacy, integrity and control over persistent context.

Architectural placement

Why this sits here

This sits in Layer 2 because memory risk materialises during runtime reads and writes, and persists beyond individual sessions.

Memory is data, context and instruction history; it must be governed like a high-risk store.
Problem statement

Agent memory can retain personal information, sensitive business context or malicious instructions over time. It may influence future actions without users realising why.

Capability description

What the capability does

  • Controls memory read and write permissions.
  • Classifies and provenance-tags memory entries.
  • Applies retention, expiry and deletion rules.
  • Quarantines or re-scores suspicious memory.
  • Records memory decisions and evidence.

What the capability does not do

  • It does not guarantee all poisoned memory is detected.
  • It does not replace data governance.
  • It does not expose hidden chain-of-thought.

Sensitive data is minimised, expired or deleted.

Poisoning resilience

Risky memory can be quarantined.

Enterprise examples

Employee assistant

Agent remembers user preferences.

Normal / desired state:
Only low-risk preferences persist with expiry.
Risk / control concern:
Agent stores health or HR details without policy.

Workflow agent

Agent remembers prior task context.

Normal / desired state:
Memory is provenance-tagged and scoped.
Risk / control concern:
A malicious instruction persists across sessions.
Failure scenarios

Toxic memory persistence

Prompt injection writes future instructions into memory.

No deletion path

User asks to delete memory but system cannot comply.

Sensitive context leak

Memory from one user influences another user session.
