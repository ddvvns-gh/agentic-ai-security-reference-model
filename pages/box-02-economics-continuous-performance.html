Measure whether AI agents are worth operating and whether they continue to deliver value safely.

This capability measures cost, utilisation, quality, productivity and business value across the AI workforce. It makes sure agents do not simply consume tokens, infrastructure and operational attention without delivering measurable outcomes.

Purpose and strategic objective

Purpose

To continuously assess whether each AI agent is delivering expected business value, quality and cost efficiency.

Strategic objective

Enable scalable AI adoption by linking agent funding, operation and continuation to measurable business outcomes and risk-adjusted value.

Architectural placement

Why this sits here

This capability sits in Layer -4 because it is a business management and portfolio governance function. Runtime FinOps controls may enforce budgets, but this layer determines whether the agent remains economically justified.

AI agents should not be allowed to become permanent cost centres without measurable value and performance accountability.
Problem statement

Agents can proliferate quickly and generate ongoing cost through model usage, API calls, cloud infrastructure, monitoring, support and human review. Without performance governance, the enterprise cannot tell which agents should be promoted, tuned, merged or retired.

Capability description

What the capability does

  • Tracks cost per agent, workflow, transaction and outcome.
  • Measures utilisation, task completion, quality and productivity.
  • Compares agent performance against expected business outcomes.
  • Supports re-certification, tuning, investment, consolidation or retirement decisions.
  • Feeds executive dashboards and portfolio management.

What the capability does not do

  • It does not enforce every runtime token or API budget directly.
  • It does not replace business-case ownership.
  • It does not guarantee that performance metrics are perfect or free from bias.
  • It does not replace operational monitoring or incident response.

Outcomes delivered

Cost transparency

Business owners understand the cost of the agents they sponsor.

Value discipline

Agents continue only when they deliver measurable value.

Performance accountability

Quality and productivity are tracked over time rather than assumed.

Enterprise examples

High-cost low-use agent

A summarisation agent uses expensive models but has low adoption.

Normal / desired state:
The scorecard triggers review and either tuning, model change or retirement.
Risk / control concern:
The agent remains active indefinitely and burns budget without value.

Customer-service automation

An agent reduces handling time but increases escalation rates.

Normal / desired state:
Performance metrics show both productivity and quality impact.
Risk / control concern:
Cost savings are celebrated while customer outcomes degrade.
Failure scenarios

Runaway operating cost

Agent usage grows with no cost owner or budget signal.

Productivity illusion

The agent appears efficient but creates rework or human correction downstream.

Unfunded support burden

Security, operations and compliance teams support agents without cost attribution.

North-star vision

Every AI agent has an economic profile and a value scorecard. Agents are promoted, tuned, restricted or retired based on evidence rather than enthusiasm.

Box 30 full-depth alignment — review incorporated

Review update: Expanded to address runaway recursive execution, token explosion and operational denial-of-service risks. Economic governance now includes runtime circuit breakers, not only retrospective cost review.

Control objective

Ensure AI economics, performance and runtime consumption are governed through real-time cost, velocity and operational safety controls that can halt runaway execution before material loss or service degradation occurs.

Control statement

Autonomous agents and high-consumption AI workloads must enforce financial and operational circuit breakers based on token velocity, reasoning-loop depth, tool invocation frequency, backend API pressure and cost per transaction block.

Expanded risk catalogue

  • R-ECO-001: AI capability consumes material cost without value accountability.
  • R-ECO-002: Cost optimisation undermines safety, resilience or assurance.
  • R-ECO-003: Performance targets are disconnected from risk and customer outcomes.
  • R-ECO-004: Economic signals drive excessive autonomy or unsafe scaling.
  • R-ECO-005: Recursive execution cascade causes rapid budget depletion.
  • R-ECO-006: Agent loop overwhelms internal backend APIs or dependent services.

Threat events

  • T-ECO-001: Autonomous workload scales without budget or policy constraints.
  • T-ECO-002: Cost-saving change removes human oversight for high-risk activity.
  • T-ECO-003: Usage grows materially without service owner accountability.
  • T-ECO-004: Performance optimisation increases response speed but reduces validation quality.
  • T-ECO-006: Recursive agent execution cascade leads to operational denial of service and rapid budget depletion.

New control: C-ECO-006 Runtime Financial Circuit Breaker

The runtime must halt or restrict an active agent execution session when cost or reasoning velocity exceeds approved thresholds for a distinct transaction block.

  • Hard limit on reasoning steps per transaction block.
  • Hard limit on token spend per transaction block.
  • Hard limit on tool invocations per time window.
  • Automatic suspension of active execution on breach.

New control: C-ECO-007 Operational Velocity Guardrail

Agent orchestration must monitor runtime loop behavior and backend service pressure. Ambiguous tool outputs, malformed database entries or repeated validation errors must trigger loop dampening, fallback handling or containment.

  • Loop-depth detection.
  • Repeated-error pattern detection.
  • API storm detection.
  • Containment and escalation workflow.

Metrics

  • M-ECO-001: Material AI workload showback coverage = 100%.
  • M-ECO-002: Cost attribution coverage ≥ 95%.
  • M-ECO-008: Cost per transaction block monitored for in-scope agents.
  • M-ECO-009: Reasoning steps per transaction block monitored for in-scope agents.
  • M-ECO-010: Circuit breaker activation rate tracked and reviewed.
  • M-ECO-011: Mean time to contain runaway execution within defined SLA.

Evidence requirements

  • E-ECO-001: FinOps dashboards.
  • E-ECO-002: Runtime cost and token telemetry.
  • E-ECO-003: Tool invocation and API pressure logs.
  • E-ECO-004: Circuit breaker activation records.
  • E-ECO-005: Containment and remediation records.
  • E-ECO-006: Post-event economic impact assessment.

Traceability

Risk: Recursive execution cascade → Threshold: velocity or cost breach → Control: Runtime Financial Circuit Breaker → Metric: reasoning steps / cost per transaction block → Evidence: circuit breaker and runtime telemetry records.
