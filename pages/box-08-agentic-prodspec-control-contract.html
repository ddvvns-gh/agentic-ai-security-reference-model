Translate governance intent into an external, machine-readable enforcement contract.

The Prodspec is the authoritative control contract consumed by gateways, identity systems, tool brokers, runtime controls, release gates and evidence systems. It does not rely on the agent to remember or obey policy; it externalises policy so other systems can enforce it.

Purpose and strategic objective

Purpose

To provide a versioned, machine-readable contract for what an agent is allowed to do, which identities it may use, what data it may access, what tools it may call and when outputs require approval.

Strategic objective

Make AI governance enforceable by external systems rather than relying on prompts or agent self-policing.

Architectural placement

Why this sits here

This capability sits in Layer -2 because it turns business governance and supply-chain trust into executable policy before SDLC and runtime enforcement begin.

The Prodspec is where governance becomes enforceable by systems outside the LLM.
Problem statement

Without a machine-readable control contract, policy remains text in documents or prompts. That creates inconsistent enforcement, self-policing risk and weak auditability.

Capability description

What the capability does

  • Defines authorised purpose, data, tools, identities, memory, cost and approval thresholds.
  • Publishes policy to external enforcement points such as AI gateways, API gateways, identity brokers and sandboxes.
  • Version-controls changes and links them to release evidence.
  • Defines which decisions are deterministic hard stops versus advisory evaluations.
  • Feeds decision lineage and assurance evidence.

What the capability does not do

  • It does not ask the model to enforce its own constraints.
  • It does not replace runtime gateways, identity brokers or policy engines.
  • It does not guarantee safe behaviour without external enforcement and monitoring.

Outcomes delivered

Externalized enforcement

Policy is consumed by control planes outside the agent.

Reduced ambiguity

Deterministic controls are separated from advisory model evaluation.

Audit clarity

Approved intent can be compared to actual behaviour.

Enterprise examples

Tool access rule

An agent is allowed to create tickets but not approve payments.

Normal / desired state:
The Prodspec publishes this rule to the tool gateway.
Risk / control concern:
The prompt says not to approve payments, but no gateway prevents it.

Memory policy

An agent may store preferences but not personal information.

Normal / desired state:
The Prodspec binds memory writes to classification and retention rules.
Risk / control concern:
The model decides for itself what is safe to remember.
Failure scenarios

Prompt-only policy

The only enforcement is a system prompt.

Unbound Prodspec

Policy exists but is not consumed by gateway controls.

Policy ambiguity

Teams cannot tell which rules are hard stops.

Control objective

Ensure the capability defines clear policy intent, technical control requirements, evidence obligations and threshold behaviour for all material AI systems that depend on this contract.

Control statement

Agentic AI systems must not rely on this capability unless the prodspec contract is versioned, approved, testable, monitored and linked to runtime evidence.

Risk catalogue

  • R-PSC-001: Prodspec contract is ambiguous or unenforceable.
  • R-PSC-002: Runtime behaviour deviates from declared contract.
  • R-PSC-003: Contract changes are not versioned, approved or tested.
  • R-PSC-004: Evidence required by contract is not produced or retained.

Threat events

  • Agent executes an action not covered by approved control contract.
  • Contract update changes permissible behaviour without revalidation.
  • Evidence collection fails during material decision or tool invocation.
  • Threshold escalation is bypassed because contract logic is incomplete.

Preventive controls

  • Contract versioning and approval.
  • Policy-as-code validation.
  • Threshold and evidence requirement declaration.
  • Runtime gating on contract validity.

Detective / corrective controls

  • Contract conformance monitoring.
  • Runtime drift detection.
  • Exception and waiver workflow.
  • Rollback to last approved contract version.

Metrics

  • Approved contract coverage = 100% for material systems.
  • Contract conformance rate ≥ 99%.
  • Unapproved contract changes = 0.
  • Evidence completeness = 100% for T3–T5 workflows.

Evidence

  • Approved prodspec contract.
  • Version history and approval record.
  • Policy-as-code test results.
  • Runtime conformance and evidence logs.

Traceability

Risk: runtime behaviour deviates from contract → Threshold: material AI workflow → Control: contract validity and conformance gate → Metric: conformance rate → Evidence: approved contract and runtime conformance logs.
