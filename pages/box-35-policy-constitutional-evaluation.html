Use evaluators as advisory validators, with deterministic policy as authority.

This capability evaluates proposed actions against enterprise policy and principles. It distinguishes deterministic policy engines from constrained evaluator models: the former enforce hard rules, while the latter provide advisory semantic assessment for ambiguous situations.

Purpose and strategic objective

Purpose

To assess whether proposed AI behaviour aligns with enterprise policy while avoiding over-reliance on probabilistic evaluators.

Strategic objective

Use dual-engine validation for high-consequence decisions: deterministic policy where possible, constrained evaluator where semantic judgement is needed, and human approval where material.

Architectural placement

Why this sits here

This sits in Layer 3 because it supports cognitive governance and accountability, not raw runtime execution.

Evaluator models can help reason about policy, but they are not the final security authority.
Problem statement

Some AI failures are logic-level failures, but using another LLM as the sole judge repeats the self-policing problem. Deterministic policy must govern hard rules, with evaluators supplementing rather than replacing enforcement.

Capability description

What the capability does

  • Evaluates actions against enterprise principles and policy.
  • Separates deterministic rules from advisory semantic evaluation.
  • Routes uncertainty, disagreement and high-impact findings to human review.
  • Version-controls evaluator instructions and policy rules.
  • Records evaluation rationale and confidence.

What the capability does not do

  • It does not treat evaluator LLM output as deterministic enforcement.
  • It does not replace tool gateways or identity controls.
  • It does not replace legal or compliance judgement.

v18 hardening — State-Locked Validation

High-impact validation and execution must operate over the same immutable state snapshot. The validation envelope should bind identity, delegated token, policy version, Prodspec version, model/prompt version, tool parameters, environment variables, data references and approval state. Execution must fail if runtime state no longer matches the signed validation envelope.

Enterprise examples

External email policy

Agent proposes sending internal data externally.

Normal / desired state:
OPA blocks prohibited domains; evaluator summarises semantic risk.
Risk / control concern:
Evaluator alone says it appears safe.

Customer decision

Agent recommends hardship outcome.

Normal / desired state:
Policy checks eligibility; evaluator checks rationale; human approves.
Risk / control concern:
One evaluator model approves final decision.
Failure scenarios

Evaluator hallucination

Evaluator misreads policy.

Hard rule omitted

Deterministic rule is only written as prompt.

Disagreement ignored

Multiple validators disagree but action proceeds.
