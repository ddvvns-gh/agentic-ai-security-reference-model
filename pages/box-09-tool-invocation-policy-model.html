Govern the tools, APIs and actions that turn agent intent into real-world effect.

This capability defines which tools an agent may use, which actions are permitted, which parameters are allowed, which destinations are acceptable and which invocations require approval.

Purpose and strategic objective

Purpose

To control the action boundary where AI agents move from generating text to interacting with systems, data and external services.

Strategic objective

Prevent tool misuse, excessive agency, unsafe API calls and unauthorised business actions by enforcing action-level policies.

Architectural placement

Why this sits here

It sits in Layer -2 because tool policy must be defined in the Prodspec before runtime agents are allowed to call tools.

Tool use is where agent risk becomes operational consequence.
Problem statement

Agents become dangerous primarily when they can call tools: send emails, query databases, change systems, approve transactions or call APIs. Coarse access controls are insufficient because risk depends on the specific action, parameters and context.

Capability description

What the capability does

  • Defines allowed tools, actions and parameters.
  • Classifies tools by impact and reversibility.
  • Sets approval and dual-control triggers.
  • Constrains destinations, scopes and rate limits.
  • Produces tool invocation evidence.

What the capability does not do

  • It does not replace tool-specific security controls.
  • It does not guarantee correct tool output.
  • It does not replace human approval for high-impact actions.

Outcomes delivered

Action-level control

Agents are constrained to specific allowed actions.

Reduced blast radius

Tool authority is limited by purpose and context.

Improved auditability

Tool calls are traceable to policy decisions.

Enterprise examples

Email tool

An agent may draft and send messages.

Normal / desired state:
Internal low-risk emails are allowed; external sensitive emails require approval.
Risk / control concern:
The agent exfiltrates documents through an allowed email tool.

Database query tool

An agent retrieves customer data for service workflows.

Normal / desired state:
Queries are restricted to approved fields and purpose.
Risk / control concern:
The agent performs broad exports outside approved use.
Failure scenarios

Allowed tool, wrong purpose

A tool is technically allowed but used for unauthorised disclosure.

Parameter abuse

A prompt injection manipulates API parameters.

Irreversible action

An agent executes a transaction without approval.
