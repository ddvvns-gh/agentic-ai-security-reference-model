Implement the Agentic Identity Mesh for task-scoped delegated authority.

This capability is the core of the Agentic Identity Mesh. It issues short-lived, task-specific credentials to agents, child agents and workloads through dynamic token exchange, preserving the lineage from initiating user to agent to sub-task to tool call.

Purpose and strategic objective

Purpose

To replace static secrets and broad inherited permissions with cryptographically bound, short-lived and task-scoped authority.

Strategic objective

Prevent credential theft, excessive agency and invisible privilege mutation across agentic workflows.

Architectural placement

Why this sits here

This sits in Layer 1 because identity and secret handling are part of the execution baseline required before any agent can act.

No identity, no token, no action.
Problem statement

Agents often act on behalf of humans across infrastructure boundaries. Simply giving the agent the human user's cumulative permissions creates excessive authority and poor accountability.

Capability description

What the capability does

  • Issues ephemeral credentials for agent tasks and sub-tasks.
  • Performs dynamic token exchange based on user, agent, task, purpose and Prodspec.
  • Preserves session chains across human, parent agent, child agent and tool call.
  • Revokes credentials on task completion, expiry, breach or suspension.
  • Records workload lineage for audit and incident response.

What the capability does not do

  • It does not grant agents all permissions of the initiating user.
  • It does not replace business approval for material actions.
  • It does not make insecure tools safe without gateway enforcement.

v18 hardening — Trust Chain integration

Agentic identity now links directly to AI provenance and supply-chain assurance. A token issued to an agent or child-agent should be bound not only to the human, agent and task, but also to the approved artefact set, Prodspec version, policy version and tool/action scope used for that task.

Outcomes delivered

Agents act only with task-specific authority.

Every action links back to human, agent and purpose.

Fast containment

Tokens can be revoked or allowed to expire rapidly.

Enterprise examples

Customer service agent

Agent needs to update a case.

Normal / desired state:
It exchanges for a token limited to the case and action.
Risk / control concern:
It receives all CRM permissions of the user.

Child agent

Parent delegates document classification.

Normal / desired state:
Child receives read-only token for specific document set.
Risk / control concern:
Child inherits full parent access.
Failure scenarios

Static API key in prompt

Secret appears in logs or model context.

Privilege overload

Agent accesses systems unrelated to the task.

Lost lineage

SOC cannot determine which user initiated a tool call.
