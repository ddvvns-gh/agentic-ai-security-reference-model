Capture intent provenance, not only execution logs.

This capability is strengthened to address the audit gap in autonomous agent chains. It captures structured semantic state: user intent, delegated authority, agent plan, tool intent, retrieved context, policy decisions, approvals and final action.

Purpose and strategic objective

Purpose

To make complex agentic decisions reconstructable, explainable and defensible without relying on raw logs alone.

Strategic objective

Provide the evidence backbone for regulated AI by capturing the why behind tool calls and commitments.

Architectural placement

Why this sits here

This sits in Layer 3 because it captures cognitive governance and accountability evidence that goes beyond infrastructure logging.

Traditional logs say what happened; agentic lineage must also preserve why the action was attempted.
Problem statement

When an agent chains 15 API calls with synthetic sub-steps, raw logs become insufficient. Investigators need structured provenance of intent and authority.

Capability description

What the capability does

  • Captures user request, agent identity, delegated authority and task purpose.
  • Records plan summary, retrieved context references and tool intent.
  • Links policy decisions, approvals, outputs and final commitments.
  • Preserves session-chain lineage across child agents and tools.
  • Stores tamper-evident evidence for audit and incident response.

What the capability does not do

  • It does not require storing hidden chain-of-thought.
  • It does not prove the decision was correct.
  • It does not replace raw telemetry or system logs.

v18 hardening — Intent, State and Semantic Provenance

Decision lineage now explicitly captures three provenance dimensions: intent provenance (why a tool call or action was attempted), state provenance (the immutable context and policy snapshot used for validation/execution), and semantic provenance (prompt, context, memory, agent-to-agent payload and output sources).

  • Intent provenance: initiating user, delegated agent, task purpose, proposed action, tool intent and commitment rationale.
  • State provenance: policy version, Prodspec version, environment variables, tool parameters, token scope, approval state and validation envelope.
  • Semantic provenance: prompt source, retrieved-context source, memory epoch, agent-to-agent payload source and output derivation trail.

Enterprise examples

15 API call workflow

Agent orchestrates multiple services.

Normal / desired state:
Evidence records task purpose, tool intent and approvals for each call.
Risk / control concern:
Logs show calls but not why they happened.

Child agent task

Parent agent delegates work.

Normal / desired state:
Lineage records parent, child, token scope and result.
Risk / control concern:
Child action appears as unrelated service account activity.
Failure scenarios

Log-only evidence

Only endpoint/API logs exist.

Missing intent

Tool call cannot be linked to business purpose.

Broken session chain

Human initiator is lost after child-agent handoff.
