Test AI systems against security, safety, data, tool and operational failure modes before release.

This capability defines the pre-production testing regime for AI agents, including red teaming, jailbreak testing, data leakage testing, tool misuse testing, bias checks, regression testing and resilience scenarios.

Purpose and strategic objective

Purpose

To identify and reduce AI-specific risk before deployment and after material changes.

Strategic objective

Ensure AI systems meet defined acceptance criteria and have signed evidence of testing, exceptions and risk acceptance before release.

Architectural placement

Why this sits here

It sits in Layer -1 because testing must occur before production readiness gates and runtime operation.

AI cannot be proven safe once and for all, but it can be tested, measured and continuously revalidated.
Problem statement

AI systems may pass functional testing while still being vulnerable to prompt injection, data leakage, tool abuse, unsafe outputs or model drift. Pre-deployment testing must address probabilistic and adversarial behaviours.

Capability description

What the capability does

  • Runs adversarial and regression tests.
  • Tests prompt injection, jailbreak and indirect injection scenarios.
  • Tests tool misuse, data leakage and unsafe output handling.
  • Records results, exceptions and acceptance criteria.
  • Triggers retesting after material changes.

What the capability does not do

  • It does not prove absence of all vulnerabilities.
  • It does not replace runtime monitoring.
  • It does not remove the need for risk acceptance.

Enterprise examples

Prompt injection suite

A customer agent is tested with direct and indirect injection prompts.

Normal / desired state:
Failures are remediated or risk accepted before release.
Risk / control concern:
The agent launches with known bypasses and no acceptance.

Tool misuse test

An agent with email and database tools is tested for exfiltration paths.

Normal / desired state:
Unsafe tool chains are blocked or routed to approval.
Risk / control concern:
Testing focuses only on answer quality, not action abuse.
Failure scenarios

False confidence

The agent passes demo tests but fails adversarial scenarios.

Untested tool path

A rarely used tool action enables data leakage.

No regression baseline

A model upgrade changes behaviour without detection.
