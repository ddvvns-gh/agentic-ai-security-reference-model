Force every tool and API action through a zero-trust gateway.

This capability defines and enforces tool-use policy outside the LLM. Every MCP interaction, API call or tool invocation passes through a deterministic gateway that validates identity, schema, parameters, destination, purpose and approval state.

Purpose and strategic objective

Purpose

To control the action boundary where agent reasoning becomes system interaction.

Strategic objective

Prevent tool misuse, excessive agency and prompt-driven API abuse by enforcing tool policy outside the agent loop.

Architectural placement

Why this sits here

This sits in Layer -2 because tool rules must be defined before runtime, but they are enforced in Layer 2 gateways and runtime controls.

The agent should never call tools directly. It should request tool use through a governed gateway.
Problem statement

Agents can be manipulated into using legitimate tools for illegitimate purposes. If the model directly controls tool execution, prompt injection can become real-world action.

Capability description

What the capability does

  • Defines allowed tools, actions, parameters and destinations.
  • Requires all tool calls to traverse a zero-trust gateway.
  • Validates schemas, parameters, mTLS identity and purpose binding.
  • Separates model suggestion from tool execution authority.
  • Records tool intent and policy decision evidence.

What the capability does not do

  • It does not rely on the LLM to decide whether tool use is safe.
  • It does not replace business approval for high-impact actions.
  • It does not make unsafe APIs safe without downstream controls.

Outcomes delivered

Deterministic action control

Tools execute only after external policy approval.

Reduced prompt injection impact

Manipulated model intent cannot directly call APIs.

Auditable tool intent

Each tool request records purpose, parameters and policy outcome.

Enterprise examples

MCP server call

Agent requests a CRM update.

Normal / desired state:
Gateway validates schema, mTLS identity, parameters and Prodspec authority.
Risk / control concern:
Agent directly calls MCP tool with manipulated parameters.

Email tool

Agent proposes external email.

Normal / desired state:
Gateway blocks sensitive attachment unless commitment approval exists.
Risk / control concern:
Model sends email because prompt says it is allowed.
Failure scenarios

Direct API path

Agent bypasses gateway and calls API directly.

Parameter smuggling

Prompt injection hides unsafe parameter in tool request.

Allowed tool wrong purpose

Tool is technically allowed but used for exfiltration.
