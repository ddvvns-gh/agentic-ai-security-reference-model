C-OCC-001

Commitment Authority Validation

Before any agent output becomes an enterprise commitment, positively verify the acting agent's delegated authority and the action's declared purpose against an authoritative source — in the control plane, not in the agent.

Control IDC-OCC-001
TypePreventive
Trust domainDecision · Identity
Enforcement pointCommitment gateway (pre-execution)
Failure postureFail-closed
Parent capabilityBox 33 · Output Commitment Control
Control objective
Ensure that no agent output is allowed to become an enterprise commitment unless the acting agent — and every principal in its delegation chain — holds current, sufficient delegated authority for that action type, value band and purpose. The objective is to make "who is allowed to commit the enterprise to what" a deterministic, externally enforced check rather than an implicit property of the model's behaviour.
Control statement
Every output classified as a commitment (per C-OCC-002) must be validated against the acting agent's and any delegating principal's current delegated authority and declared purpose before execution. Validation must be performed by the external control plane, must fail closed on any missing authority, stale matrix or unresolved delegation lineage, and must emit an authority-validation record for every decision.
Design & operation
  1. Trigger. An output is classified as commitment-bearing by C-OCC-002 (e.g. payment, contract, external representation, irreversible state change).
  2. Resolve principal & lineage. The gateway resolves the agent's cryptographic identity and full delegation chain from the identity/lineage controls (Box 25), including any human-on-behalf-of relationship.
  3. Look up authority. The action type, counterparty class, value band and declared purpose are evaluated against the versioned Delegated Authority Framework and Commitment Threshold Matrix.
  4. Decide. If the action is wholly within authority → allow. If it exceeds authority or purpose → deny and route to human approval (C-OCC-004) or escalation; threshold breaches invoke C-OCC-008.
  5. Record. The decision, inputs and matched authority are written as an Authority Validation Record (E-OCC-001).
This is a deterministic, external control: the agent cannot self-attest its own authority, and a non-response or error resolves to deny.
Inputs / outputs

Inputs

  • Agent identity + delegation lineage (Box 25)
  • Commitment classification (C-OCC-002)
  • Action parameters: type, counterparty, value band
  • Declared purpose / Prodspec binding (Box 08)
  • Delegated-authority matrix + threshold matrix (versioned)

Outputs

  • Allow / deny decision (pre-execution)
  • Authority Validation Record (E-OCC-001)
  • On deny: approval trigger (C-OCC-004) / escalation
  • Decision feed to lineage (C-OCC-005) and monitoring (C-OCC-006)

Test method / assurance

Design effectiveness

  • Confirm the authority + threshold matrices are versioned, owned and cover all in-scope commitment types.
  • Confirm the gateway is the sole commitment path and is configured fail-closed.
  • Confirm delegation-lineage resolution is mandatory (no anonymous principals).

Operating effectiveness

  • Sample executed commitments; confirm each has a matching E-OCC-001 with authority that covers the action.
  • Inject an out-of-authority action in the shadow environment (Box 15); confirm deny + record + escalation.
  • Reconcile M-OCC-001 spikes to investigations.
Failure modes & residual risk
  • Classification miss. If C-OCC-002 fails to flag an output as a commitment, this control is never invoked — residual risk is inherited from classification coverage (M-OCC-006 registration coverage).
  • Stale authority data. An out-of-date matrix may permit a no-longer-valid action; mitigated by versioning + recertification of the Delegated Authority Framework.
  • Lineage gaps. Incomplete delegation lineage from Box 25 forces fail-closed (safe) but can create friction; tracked via M-OCC-004 lineage completeness.
  • Residual. With C-OCC-002 coverage high and Box 25 lineage complete, residual risk for R-OCC-001 is Low; degrades to Medium where classification or lineage coverage is partial.
Regulatory mapping
  • APRA CPS 230 — accountability and operational-risk control over automated decisions that create obligations.
  • APRA CPS 234 — design and operating-effectiveness evidence for an information-security control.
  • Corporations Act / ASIC conduct — authority to bind the entity in customer-affecting commitments.
  • NIST AI RMF (Manage) / ISO 42001 — controlled, accountable operation of an AI system.
Implementation notes (vendor-neutral)
Implement as a policy-as-code rule in the commitment gateway, evaluating signed identity/lineage claims against the authority matrix held as versioned data. Integrate with enterprise IAM/entitlements for the human-on-behalf-of leg, and bind the agent's permitted commitment scope in its Prodspec (Box 08) so authority is declared at design time and enforced at runtime. Technology is illustrative; the control is defined by the objective, statement and evidence — not a product.
