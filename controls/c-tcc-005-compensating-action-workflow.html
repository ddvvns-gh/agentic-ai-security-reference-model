C-TCC-005

Compensating-Action Workflow

For effects that cannot be rolled back, execute approved compensating actions to reverse or offset the harm — and reconcile dependent-system state afterwards.

Control IDC-TCC-005
TypeCorrective
Trust domainDecision · Execution
Enforcement pointWorkflow / case management
Failure postureTracked to closure
Parent capabilityBox 34 · Terminate, Contain & Compensate — Hard
Control objective
Address the reality that many agent actions are irreversible (you cannot un-send a payment or a statement). Where containment cannot undo an effect, this control drives the compensating actions that reverse or offset it and reconciles the resulting state across dependent systems.
Control statement
Where a contained agent has produced irreversible effects, a compensating-action workflow must be initiated, assigned an accountable owner, and tracked to completion, with dependent-system state reconciled; compensation must be evidenced.
Design & operation
  1. Identify irreversible effects from the kill/execution records and the commitment register (Box 33).
  2. Select the approved compensating action per effect type (recall/return, correction, counterparty notification, ledger adjustment).
  3. Execute under approval and an accountable owner; reconcile dependent-system state.
  4. Track to closure within SLA; record actions and outcomes (E-TCC-004).
Inputs / outputs

Inputs

  • Irreversible-effect inventory
  • Compensating-action playbooks
  • Owner + approval

Outputs

  • Compensating actions executed
  • State reconciliation
  • Compensation Record (E-TCC-004)

Test method / assurance

Design effectiveness

  • Confirm playbooks exist for the main irreversible effect types.
  • Confirm each compensation has an owner and SLA.
  • Confirm dependent-system reconciliation is in scope (backlog item 12).

Operating effectiveness

  • Tabletop an irreversible-effect scenario; confirm compensation completes within SLA.
  • Measure compensation completion (M-TCC-004).
  • Confirm reconciliation closed the dependent-system gap.
Failure modes & residual risk
  • No reconciliation. Compensating the primary effect but not dependent systems leaves residual inconsistency — the model's known under-specified area.
  • Residual. Medium; this is inherently a partial, after-the-fact control.
Regulatory mapping
  • APRA CPS 230 — recovery and operational resilience.
  • Conduct obligations — customer remediation.
  • ISO 42001 — corrective action.
Implementation notes (vendor-neutral)
Implement as playbook-driven case management tied to the commitment register (Box 33) and ledgers; explicitly include dependent-system reconciliation. Addresses the reviewer's recovery-economics gap (backlog item 12).
